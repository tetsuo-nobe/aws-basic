# AWS CloudFormation ハンズオン

AWS CloudFormation を使用して、Web アプリケーション基盤（VPC、EC2、RDS、ALB）を一括で構築するハンズオンです。テンプレートファイルからスタックを作成し、構築されたリソースを確認した後、クリーンアップするまでの一連の流れを体験します。

## 前提条件

- AWS アカウントを持っていること
- インストラクターのガイドに基づき AWS マネジメントコンソールにサインインします。
- 今回は **東京 (ap-northeast-1)** を使用します。

---

## テンプレートの概要

使用するテンプレート `webapp_template.yaml` は、以下のリソースを作成します。

| カテゴリ | リソース | 説明 |
|---------|---------|------|
| ネットワーク | VPC | CIDR `10.0.0.0/16` の VPC |
| ネットワーク | パブリックサブネット × 2 | ALB を配置（2 つの AZ に分散） |
| ネットワーク | プライベートサブネット × 2 | EC2 と RDS を配置（2 つの AZ に分散） |
| ネットワーク | インターネットゲートウェイ | パブリックサブネットからインターネットへの通信用 |
| ネットワーク | NAT Gateway | プライベートサブネットからインターネットへのアウトバウンド通信用 |
| ネットワーク | ルートテーブル × 2 | パブリック用とプライベート用 |
| セキュリティ | セキュリティグループ × 3 | ALB 用、AP サーバー用、DB サーバー用 |
| コンピューティング | EC2 インスタンス | Apache HTTP Server がインストールされた Web サーバー |
| データベース | RDS (MariaDB) | MariaDB 10.11 のデータベースインスタンス |
| ロードバランサー | ALB | インターネットからのリクエストを EC2 に振り分ける |
| IAM | IAM ロール | Session Manager 接続用のロール |

## 1. テンプレートの内容を確認する

スタックを作成する前に、テンプレートの構成を理解しましょう。

### 1-1. テンプレートの主要セクション

`webapp_template.yaml` を開き、以下のセクションを確認します。

| セクション | 説明 |
|-----------|------|
| `AWSTemplateFormatVersion` | テンプレートのバージョン（`2010-09-09` が現在の唯一のバージョン） |
| `Description` | テンプレートの説明文 |
| `Parameters` | スタック作成時にユーザーが入力できるパラメータ |
| `Mappings` | リージョンごとの AMI ID のマッピング |
| `Resources` | 作成する AWS リソースの定義 |
| `Outputs` | スタック作成後に表示する出力値 |

### 1-2. パラメータの確認

テンプレートには以下のパラメータが定義されています。

| パラメータ名 | デフォルト値 | 説明 |
|-------------|-------------|------|
| `VPCCIDR` | `10.0.0.0/16` | VPC の CIDR ブロック |
| `PUBLICSUBNET01CIDR` | `10.0.0.0/24` | パブリックサブネット 1 の CIDR |
| `PUBLICSUBNET02CIDR` | `10.0.1.0/24` | パブリックサブネット 2 の CIDR |
| `PRIVATESUBNET01CIDR` | `10.0.2.0/24` | プライベートサブネット 1 の CIDR |
| `PRIVATESUBNET02CIDR` | `10.0.3.0/24` | プライベートサブネット 2 の CIDR |
| `DBADMINUSER` | `root` | データベースの管理者ユーザー名 |
| `DBADMINPASSWORD` | `rootpassword` | データベースの管理者パスワード |

> ハンズオンではデフォルト値のまま使用しますが、本番環境では必ずパスワードを変更してください。

## 2. スタックの作成

### 2-1. CloudFormation コンソールを開く

1. AWS マネジメントコンソールにサインインする
2. リージョンが **東京 (ap-northeast-1)** であることを確認する
3. 上部の検索バーに「CloudFormation」と入力し、CloudFormation サービスを選択する

### 2-2. スタックを作成する

1. 左メニューから「スタック」をクリックする
2. 「スタックの作成」から「新しいリソースを使用（標準）」を選択する

### 2-3. テンプレートの指定

1. 「テンプレートの準備」で「既存のテンプレートを選択」を選択する
2. 「テンプレートソース」で「テンプレートファイルのアップロード」を選択する
3. 「ファイルの選択」をクリックし、`webapp_template.yaml` をアップロードする
4. 「次へ」をクリックする

### 2-4. スタックの詳細を指定

1. 「スタック名」に `webapp-stack` と入力する（任意の名前でも可）
2. パラメータはすべてデフォルト値のまま「次へ」をクリックする

> パラメータの値はテンプレートで定義されたデフォルト値が自動的に入力されています。

### 2-5. スタックオプションの設定

1. この画面ではデフォルトのまま変更しない
2. 「次へ」をクリックする

### 2-6. 確認と作成

1. 設定内容を確認する
2. ページ下部の「AWS CloudFormation によって IAM リソースがカスタム名で作成される場合があることを承認します。」に**チェックを入れる**
3. 「次へ」をクリックする

> このテンプレートは IAM ロール（`myshopRole`）を作成するため、IAM リソースの作成を承認する必要があります。

4. ページの右下にある「送信」をクリックする
   
### 2-7. スタックの作成を待つ

1. 「スタックの情報」タブをクリックする
1. スタックのステータスが `CREATE_IN_PROGRESS` になっていることを確認する
1. 「イベント」タブで各リソースの作成状況をリアルタイムに確認できる
1. 右上の更新ボタン（🔄）を定期的にクリックして進捗を確認する
1. ステータスが **`CREATE_COMPLETE`** になるまで待つ（約 10〜15 分）

> RDS インスタンスの作成に最も時間がかかります。

## 3. 作成されたリソースの確認

### 3-1. スタックの出力を確認する

1. CloudFormation コンソールで `webapp-stack` を選択する
2. 「出力」タブをクリックする
3. 以下の出力値を確認する

| キー | 説明 |
|------|------|
| `DBInstanceEndpoint` | RDS データベースのエンドポイント |
| `APServerInstanceId` | EC2 インスタンスの ID |
| `ALBDNSName` | ALB の DNS 名（`http://` 付き） |

### 3-2. ALB 経由で Web ページにアクセスする

1. 「出力」タブの `ALBDNSName` の値（URL）をコピーする
2. ブラウザの新しいタブを開く
3. コピーした URL を貼り付けてアクセスする
4. 「Welcome to AWS at ...」というページが表示されれば成功

> EC2 インスタンスの起動直後はヘルスチェックが完了していない場合があります。`503` エラーが表示された場合は、1〜2 分待ってから再度アクセスしてください。

### 3-3. 作成されたリソースを確認する

「リソース」タブをクリックすると、テンプレートで定義したすべてのリソースとその物理 ID を確認できます。各リソースのリンクをクリックすると、該当サービスのコンソール画面に遷移できます。

以下のリソースを確認してみましょう。

#### VPC の確認

1. 「リソース」タブで `VPC` の物理 ID のリンクをクリックする
2. VPC コンソールで `myshopVPC` が作成されていることを確認する
3. CIDR ブロックが `10.0.0.0/16` であることを確認する

#### EC2 インスタンスの確認

1. 「リソース」タブで `APServer` の物理 ID のリンクをクリックする
2. EC2 コンソールで `myshop ap` インスタンスが実行中であることを確認する
3. プライベートサブネットに配置されていること（パブリック IP がないこと）を確認する

#### RDS の確認

1. 「リソース」タブで `DBServer` の物理 ID のリンクをクリックする
2. RDS コンソールで `myshopdb` インスタンスが利用可能であることを確認する
3. エンジンが MariaDB であることを確認する

## 4. クリーンアップ

ハンズオン終了後、不要な課金を防ぐためにスタックを削除します。

> CloudFormation でスタックを削除すると、スタックで作成されたすべてのリソースが自動的に削除されます。リソースを 1 つずつ手動で削除する必要はありません。これが CloudFormation の大きなメリットの 1 つです。

### 4-1. スタックを削除する

1. CloudFormation コンソールで `webapp-stack` を選択する
2. 「スタックを削除」ボタンをクリックする
3. 確認ダイアログでスタック名を入力して「スタックを削除」をクリックする
4. ステータスが `DELETE_IN_PROGRESS` になることを確認する
5. スタックが完全に削除されるまで待つ（約 10〜15 分）

> 削除中に「イベント」タブで各リソースの削除状況を確認できます。RDS インスタンスの削除に最も時間がかかります。

### 4-2. 削除の完了を確認する

1. スタック一覧から `webapp-stack` が消えていることを確認する
   - 「削除済み」フィルターを有効にすると、削除されたスタックの履歴を確認できる

## まとめ

このハンズオンで学んだことを振り返りましょう。

| 学んだこと | 内容 |
|-----------|------|
| テンプレート | YAML 形式でインフラ構成を定義できる |
| パラメータ | スタック作成時に値を動的に指定できる |
| マッピング | リージョンごとに異なる値（AMI ID など）を定義できる |
| スタック作成 | テンプレートから複数のリソースを一括で作成できる |
| 出力 | 作成されたリソースの情報（エンドポイント、URL など）を確認できる |
| スタック更新 | テンプレートを変更してスタックを更新できる |
| スタック削除 | スタックを削除すると関連リソースがすべて削除される |

> CloudFormation を使うことで、インフラ構成をコードとして管理（Infrastructure as Code）でき、環境の再現性や一貫性を確保できます。
